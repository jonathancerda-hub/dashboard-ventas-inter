Proyecto Dashboard de Ventas Farmacéuticas
Versión 1.0
Fecha Diciembre 2025
Estado Activo

1. Resumen Ejecutivo

El Dashboard de Ventas Internacional es una aplicación web desarrollada en Flask que funciona como sistema de inteligencia de negocios para el área de ventas internacionales de una empresa farmacéutica. La plataforma se conecta en tiempo real con el sistema ERP Odoo y Supabase para visualizar, analizar y gestionar datos de ventas, pedidos y metas comerciales.

Objetivos del Proyecto

  • Proporcionar visibilidad en tiempo real del desempeño comercial internacional
  • Facilitar el seguimiento de metas de venta por cliente y línea comercial
  • Automatizar el cálculo de KPIs y métricas comerciales
  • Simplificar el análisis de datos mediante visualizaciones interactivas
  • Reducir el tiempo de generación de reportes comerciales

2. Problema a Resolver

El equipo de ventas internacionales enfrentaba desafíos significativos en la gestión y análisis de información comercial:

  • Dispersión de Datos: Información fragmentada entre Odoo, hojas de cálculo y reportes manuales
  • Falta de Visibilidad: Dificultad para obtener una vista consolidada del desempeño por cliente
  • Procesos Manuales: Generación manual de reportes que consume tiempo valioso del equipo comercial
  • Seguimiento de Metas: Ausencia de un sistema unificado para comparar ventas reales vs objetivos
  • Análisis Limitado: Carencia de herramientas para drill-down y análisis multidimensional

3. Solución Propuesta

Dashboard web centralizado que integra múltiples fuentes de datos y proporciona análisis visual en tiempo real del desempeño comercial internacional.

3.1 Arquitectura del Sistema

Backend

  • Python 3.x - Lenguaje de programación principal
  • Flask - Framework web ligero
  • xmlrpc.client - Comunicación con API Odoo
  • Pandas - Manipulación y análisis de datos
  • supabase - Integración con Supabase (PostgreSQL)

Frontend

  • HTML5 & Jinja2 - Estructura y templates
  • CSS3 - Estilos personalizados
  • JavaScript - Interactividad
  • ECharts - Gráficos interactivos avanzados
  • Chart.js - Visualizaciones adicionales

Fuentes de Datos

  • Odoo ERP - Datos de ventas, pedidos, clientes
  • Supabase (PostgreSQL) - Gestión de metas y equipos
  • XML-RPC API - Protocolo de integración

3.2 Flujo de Datos

Petición del Usuario: Navegación a una URL específica (ej. /dashboard)
Autenticación: Verificación de sesión activa contra usuarios de Odoo
Recolección de Datos: OdooManager obtiene ventas y pendientes; SupabaseManager obtiene metas
Procesamiento: Agregación, cálculo de KPIs y preparación de datasets para gráficos
Renderizado: Generación de HTML con datos inyectados via Jinja2
Visualización: JavaScript carga gráficos interactivos en el navegador

4. Funcionalidades Principales

🔐 Autenticación Segura

Sistema de login/logout integrado con base de datos de usuarios de Odoo. Control de acceso basado en whitelist de emails autorizados.

📊 Dashboard Internacional

Vista principal con KPIs en tiempo real: ventas totales, meta anual, brecha comercial, avance porcentual. Filtrado dinámico por cliente.

📈 Análisis por Línea Comercial

Tabla consolidada mostrando ventas, pendientes, metas y % de avance por cada línea comercial internacional (AGROVET, PETMEDICA, AVIVET).

🎯 Seguimiento por Cliente

Tarjeta de avance individual con barra de progreso y semáforo (rojo/amarillo/verde). Gráfico detallado del avance por pedido de venta.

🔍 Drill-Down Jerárquico

Exploración interactiva de ventas: Clasificación Farmacológica → Forma Farmacéutica → Vía Administración → Línea Producción → Producto.

📋 Vistas Detalladas

Tablas paginadas de ventas facturadas (/sales) y pedidos pendientes (/pending) con filtros avanzados por fecha, cliente y búsqueda de texto.

📥 Exportación a Excel

Generación de reportes Excel con formato profesional para ventas facturadas y pendientes, incluyendo 17 columnas de información detallada.

🎯 Gestión de Metas

Interfaces para configurar metas anuales por cliente y línea comercial. Integración con Supabase para persistencia y consultas SQL.

👥 Dashboard por Línea (Nacional)

Vista especializada para análisis de ventas nacionales por equipo comercial, con seguimiento de metas mensuales por vendedor.

⚡ Caché Inteligente

Sistema de caché en memoria (SimpleCache) para reducir consultas a Odoo y mejorar tiempos de respuesta en dashboard principal.

🌐 Contexto Multiidioma

Soporte para español (PE) en lectura de datos desde Odoo, asegurando nombres correctos de productos sin sufijos de traducción.

💰 Manejo de Divisas

Cálculos en USD ($) usando amount_currency. Formateo automático con separadores de miles y símbolos de moneda apropiados.

6. Requisitos Técnicos

6.1 Requisitos Funcionales

  • El sistema DEBE autenticar usuarios contra la base de datos de Odoo
  • El sistema DEBE filtrar solo usuarios autorizados mediante whitelist
  • El dashboard DEBE mostrar datos en tiempo real (máximo 5 minutos de caché)
  • Los filtros de cliente DEBEN recalcular todos los KPIs y gráficos dinámicamente
  • Las exportaciones a Excel DEBEN incluir los 17 campos especificados
  • Los gráficos DEBEN ser interactivos y permitir drill-down
  • El sistema DEBE soportar paginación en vistas de tablas (configurable: 100, 500, 1000 registros)
  • Las metas DEBEN poder configurarse por año/mes y persistirse en Supabase

6.2 Requisitos No Funcionales

  • Rendimiento: Tiempo de carga del dashboard < 3 segundos con caché
  • Escalabilidad: Soporte para mínimo 50 usuarios simultáneos
  • Disponibilidad: 99% uptime durante horario laboral
  • Seguridad: Sesiones con timeout de 24 horas, credenciales en variables de entorno
  • Usabilidad: Interfaz intuitiva sin necesidad de capacitación formal
  • Compatibilidad: Soporte para Chrome, Firefox, Edge (últimas 2 versiones)
  • Mantenibilidad: Código documentado, separación de capas (MVC)

6.3 Configuración del Entorno

Variables de entorno requeridas en archivo .env:

  • ODOO_URL - URL del servidor Odoo
  • ODOO_DB - Nombre de la base de datos
  • ODOO_USER - Usuario con permisos de lectura
  • ODOO_PASSWORD - Contraseña del usuario
  • SECRET_KEY - Clave secreta para sesiones Flask
  • SUPABASE_URL - URL del proyecto Supabase
  • SUPABASE_KEY - API Key de Supabase

7. Modelos de Datos Clave

7.1 Estructura de Ventas (sales_data)

Registro procesado de líneas de factura desde Odoo:

  • Información del pedido: pedido, factura, fecha, mes
  • Información del cliente: cliente, país, partner_id
  • Información del producto: código, nombre, descripción, línea comercial
  • Clasificaciones: farmacológica, forma farmacéutica, vía administración, línea producción
  • Valores comerciales: cantidad_facturada, precio_unitario, total, amount_currency

7.2 Estructura de Pendientes (pending_data)

Registro de líneas de pedidos no facturadas desde sale.order.line:

  • Similar a sales_data pero con cantidad_pendiente y total_pendiente
  • Campo qty_to_invoice para determinar unidades no facturadas
  • Incluye descuentos aplicados a la línea

7.3 Estructura de Metas (Supabase)

Tablas principales en PostgreSQL:

  • metas_clientes: Metas anuales por cliente
  • metas_lineas: Metas mensuales por línea comercial
  • equipos: Composición de equipos comerciales
  • metas_vendedores: Metas individuales por vendedor

Dos pestañas principales:

  • Metas por Cliente: {año: {cliente_id: {linea_id: monto}}}
  • Metas por Línea: {mes: {linea_id: {meta: monto, meta_ipn: monto}}}

8. Integraciones

8.1 Integración con Odoo (XML-RPC)

Modelos consultados:

  • account.move.line - Líneas de facturas (ventas facturadas)
  • account.move - Cabecera de facturas (datos de factura, cliente, vendedor)
  • sale.order - Pedidos de venta (para mapeo y totales)
  • sale.order.line - Líneas de pedidos (pendientes de facturar)
  • product.product - Productos (atributos farmacéuticos)
  • res.partner - Clientes (datos de contacto y país)
  • res.users - Usuarios (autenticación)

Dominios de filtrado aplicados:

  • Diario: move_id.journal_id.name = 'F150 (Venta exterior)'
  • Estado: move_id.state = 'posted'
  • Cuenta: account_id.code like '70%'
  • Impuesto: tax_ids.name ilike 'EXE_IGV_EXP'
  • Exclusiones: productos con código SERV, códigos que inician con 81

8.2 Integración con Supabase

Librería: supabase-py

Autenticación: API Key y URL del proyecto

Operaciones:

  • Lectura de metas por cliente (pestaña por año)
  • Lectura de metas por línea (pestaña por mes)
  • Escritura de metas desde interfaces web
  • Gestión de equipos comerciales por línea

9. Consideraciones de Seguridad

9.1 Autenticación y Autorización

  • OAuth 2.0 de Google: Autenticación delegada a Google Identity Platform
  • Sin contraseñas locales: No se almacenan ni manejan contraseñas directamente
  • Whitelist de usuarios: Doble verificación (OAuth + allowed_users.json)
  • Tokens verificados: Verificación criptográfica de id_token de Google
  • CSRF Protection: Parámetro state en flujo OAuth para prevenir ataques cross-site

9.2 Protección OWASP Top 10 (2021)

La aplicación implementa controles de seguridad que mitigan las principales amenazas identificadas por OWASP:

  • A01 - Broken Access Control: OAuth 2.0 + whitelist + verificación de sesión en todas las rutas
  • A02 - Cryptographic Failures: HTTPS obligatorio, verificación de tokens, SECRET_KEY fuerte
  • A03 - Injection: Consultas parametrizadas, escape automático de templates Jinja2
  • A04 - Insecure Design: Uso de estándares OAuth 2.0, separación de responsabilidades
  • A05 - Security Misconfiguration: Credenciales en .env, scopes mínimos, debug off en producción
  • A07 - Authentication Failures: MFA heredado de Google, detección de anomalías, session management seguro

9.3 Gestión de Sesiones

  • Cookies seguras: HttpOnly, SameSite=Lax, firmadas con SECRET_KEY
  • Timeout: Sesiones con duración limitada (24 horas)
  • Logout completo: session.clear() invalida toda la sesión

9.4 Comunicación y Datos

  • HTTPS: Requerido en producción para OAuth 2.0 y transmisión de datos
  • Credenciales: Variables de entorno (.env), nunca en código fuente
  • Validación: Sanitización de inputs en filtros y búsquedas
  • Logs: Nivel INFO para operaciones, ERROR para excepciones (sin credenciales en logs)

10. Despliegue y Mantenimiento

10.1 Requisitos del Servidor

  • Python 3.8 o superior
  • Acceso de red a servidor Odoo (puerto según configuración)
  • Acceso a internet para Supabase API
  • 4GB RAM mínimo (8GB recomendado)
  • 2 CPU cores mínimo

10.2 Instalación

  1. Clonar repositorio del proyecto
  2. Crear entorno virtual Python: python -m venv venv
  3. Activar entorno: venv\Scripts\activate.ps1 (Windows) o source venv/bin/activate (Linux)
  4. Instalar dependencias: pip install -r requirements.txt
  5. Configurar archivo .env con credenciales
  6. Configurar variables de entorno SUPABASE_URL y SUPABASE_KEY
  7. Ejecutar aplicación: python app.py

10.3 Mantenimiento Periódico

  • Semanal: Revisión de logs de errores
  • Mensual: Actualización de metas en Supabase
  • Trimestral: Actualización de dependencias Python (pip upgrade)
  • Anual: Revisión de lista de usuarios autorizados

13. Riesgos y Mitigaciones

Riesgo Probabilidad Impacto Mitigación
Caída del servidor Odoo Media Alto Sistema de caché, mensaje de error amigable, retry automático
Cambios en API de Odoo Baja Alto Capa de abstracción (OdooManager), pruebas regulares
Rendimiento con gran volumen Media Medio Optimización de queries, índices en Odoo, caché agresivo
Inconsistencia de datos Baja Alto Validaciones en backend, logs detallados, auditorías periódicas
Pérdida de credenciales Baja Alto Backup de .env, rotación periódica, acceso restringido
