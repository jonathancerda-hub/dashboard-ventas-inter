Manual del Desarrollador

Introducción

Este manual describe la arquitectura y el flujo de datos de las secciones más importantes de la aplicación: las vistas de Ventas Facturadas (/sales) y Pedidos Pendientes (/pending).

El objetivo es proporcionar una guía técnica para entender cómo se obtienen los datos desde Odoo, cómo se procesan en el backend de Flask y cómo se presentan en el frontend.

Arquitectura del Proyecto

La aplicación sigue una arquitectura de capas bien definida, lo que facilita su mantenimiento y escalabilidad. Las responsabilidades están claramente separadas en las siguientes capas:

Capa de Lógica de Negocio (app.py)

Este archivo es el cerebro de la aplicación. Orquesta todas las operaciones:

  • Define todas las rutas (URLs) de la aplicación Flask.
  • Maneja las peticiones HTTP (GET, POST) de los usuarios.
  • Coordina las llamadas a las capas de acceso a datos (los "managers") para obtener la información necesaria.
  • Procesa, agrega y transforma los datos crudos en estructuras listas para ser visualizadas (KPIs, datos para gráficos, etc.).
  • Pasa los datos procesados a la capa de presentación para su renderizado.

Capa de Acceso a Datos (Managers)

Esta capa abstrae la complejidad de la comunicación con las fuentes de datos externas.

  • odoo_manager.py: Encapsula toda la lógica para conectarse y consultar la API XML-RPC de Odoo. Contiene métodos específicos como get_sales_lines o get_pending_orders que devuelven datos de Odoo en un formato estructurado.
  • supabase_manager.py: Se encarga de la interacción con la base de datos PostgreSQL en Supabase. Proporciona métodos para leer y escribir las metas de venta y la configuración de los equipos comerciales.

Capa de Presentación (templates/)

Contiene todos los archivos HTML que el usuario ve. Utiliza el motor de plantillas Jinja2 para renderizar dinámicamente el contenido.

  • Recibe los datos procesados desde app.py.
  • Utiliza bucles, condicionales y filtros de Jinja2 para mostrar la información.
  • Inyecta datos en formato JSON dentro de etiquetas <script> para que sean consumidos por las librerías de gráficos (ECharts, Chart.js).

Archivos Estáticos y Configuración

  • static/: Almacena archivos CSS, JavaScript del lado del cliente e imágenes.
  • .env: Archivo crucial que almacena todas las credenciales y configuraciones sensibles (URLs, contraseñas, claves de API) fuera del código fuente, garantizando la seguridad.

Dependencias

Las dependencias del proyecto se gestionan a través de un archivo requirements.txt (no incluido en el contexto, pero es una práctica estándar). A continuación se listan las tecnologías y librerías más importantes.

Backend

  • Python: Lenguaje principal.
  • Flask: Micro-framework web para la estructura de la aplicación.
  • xmlrpc.client: Librería nativa de Python para la comunicación con la API de Odoo.
  • supabase: Para la interacción con la base de datos PostgreSQL en Supabase.
  • Pandas: Utilizado para la manipulación avanzada y agregación de datos.
  • python-dotenv: Para cargar las variables de entorno desde el archivo .env.
  • openpyxl: Para generar los archivos de exportación en formato .xlsx.
  • Flask-Caching: Para implementar el sistema de caché en memoria.

Frontend

  • HTML5 & Jinja2: Para la estructura y el renderizado dinámico de las páginas.
  • CSS3: Para el diseño y la apariencia visual.
  • JavaScript: Para la interactividad del cliente (gráficos, eventos).
  • ECharts & Chart.js: Librerías de visualización de datos.

Flujo de Datos: Ventas (/sales)

Esta sección muestra un listado paginado y filtrable de todas las líneas de facturas de venta que pertenecen al canal internacional.

1. Interfaz de Usuario (templates/sales.html)

La interfaz contiene un formulario de filtros que envía los parámetros al backend.

  • Método de Envío: El formulario utiliza method="POST" para enviar los filtros. La paginación funciona con parámetros GET en la URL (ej. ?page=2).
  • Campos de Filtro: Se incluyen campos para rango de fechas (date_from, date_to), un término de búsqueda general (search_term) y un selector de clientes (partner_id).
  • Renderizado: Los datos recibidos del backend en la variable sales_data se iteran para construir las filas de la tabla. El objeto pagination se usa para generar los controles de paginación.

2. Backend (app.py - ruta /sales)

La función sales() orquesta todo el proceso en el servidor.

  1. Recepción de Filtros: Lee los parámetros de filtro desde request.form (para POST) o request.args (para GET). Esto permite que los filtros persistan al cambiar de página.
  2. Llamada al Manager: Invoca a data_manager.get_sales_lines(), pasando los filtros y los parámetros de paginación (page, per_page).
  3. Manejo de Paginación: Recibe una tupla (datos, paginacion) del manager. El objeto pagination contiene el total de registros, número de páginas, etc.
  4. Renderizado: Pasa los datos de ventas filtrados y el objeto de paginación a la plantilla sales.html.

3. Capa de Datos (odoo_manager.py - método get_sales_lines)

Este es el núcleo de la extracción de datos. Se conecta a Odoo y construye una consulta compleja.

Modelo Principal de Odoo

La consulta se realiza sobre el modelo account.move.line (líneas de asiento contable).

Construcción del Dominio (Filtros)

El dominio de la consulta se construye combinando filtros estáticos y dinámicos:

  • Filtros Estáticos (Hardcoded):
    • ('move_id.journal_id.name', '=', 'F150 (Venta exterior)'): Asegura que solo se obtengan facturas del diario de ventas internacionales.
    • ('move_id.state', '=', 'posted'): Solo facturas confirmadas.
    • ('account_id.code', '=like', '70%'): Solo líneas que afectan a cuentas de ingresos (que empiezan con 70).
    • ('tax_ids.name', 'ilike', 'EXE_IGV_EXP'): Filtra por el impuesto específico de exportación.
    • ('product_id', '!=', False): Excluye líneas que no están asociadas a un producto.
  • Filtros Dinámicos (del usuario):
    • Fechas: Se añaden condiciones ('move_id.invoice_date', '>=', date_from) y ('move_id.invoice_date', '<=', date_to).
    • Cliente: Se añade ('move_id.partner_id', '=', partner_id).
    • Búsqueda: Se construye una condición OR compleja para buscar el término search en el nombre de la factura, origen, nombre/código del producto y nombre del cliente.

Campos Obtenidos de Odoo

La consulta principal a account.move.line solicita campos clave como:

'move_id', 'partner_id', 'product_id', 'balance', 'move_name', 'quantity', 'price_unit', 'amount_currency', 'display_name'
Idioma: Todas las consultas que extraen nombres de campos (como productos o clientes) incluyen 'context': {'lang': 'es_PE'} para obtener las traducciones en español directamente desde Odoo.

Obtención de Datos Relacionados

Después de obtener las líneas base, se realizan consultas adicionales para enriquecer los datos, usando los IDs recolectados:

Búsqueda Inversa de Pedidos: Para obtener el número de pedido de venta de forma fiable, el sistema realiza una "búsqueda hacia atrás". Busca en `sale.order.line` las líneas que generaron las `account.move.line` obtenidas. Esto es más robusto que depender del campo de texto `invoice_origin`.
  1. Facturas (account.move): Se obtienen detalles de las facturas (fecha, origen, vendedor, etc.).
  2. Productos (product.product): Se obtienen todos los detalles del producto, incluyendo display_name, línea comercial, clasificación farmacológica, etc.
  3. Clientes (res.partner): Se obtienen el nombre y país del cliente.
  4. Pedidos (sale.order): Si la factura tiene un pedido de origen, se obtienen sus detalles.

Estructura Final

Finalmente, se ensambla una lista de diccionarios, donde cada diccionario representa una línea de venta con todos los datos combinados, lista para ser mostrada en la tabla.

Flujo de Datos: Pendientes (/pending)

Esta sección es muy similar a /sales, pero se enfoca en mostrar las líneas de pedidos de venta que aún no han sido completamente facturadas.

1. Interfaz y Backend (templates/pending.html y app.py)

El flujo es prácticamente idéntico al de la vista de ventas: el formulario envía los filtros, la ruta pending() en app.py los recibe y llama a data_manager.get_pending_orders(), pasando el resultado a la plantilla.

2. Capa de Datos (odoo_manager.py - método get_pending_orders)

La diferencia fundamental radica en el modelo y los filtros utilizados en Odoo.

Modelo Principal de Odoo

La consulta se realiza sobre el modelo sale.order.line (líneas de pedido de venta).

Construcción del Dominio (Filtros)

  • Filtros Estáticos (Hardcoded):
    • ('order_id.team_id.name', 'ilike', 'INTERNACIONAL'): Filtra por pedidos asignados al equipo de ventas internacional.
    • ('order_id.state', 'in', ['credit', 'sale', 'done']): Solo pedidos en estados relevantes (aprobados para crédito, confirmados o finalizados pero con saldo pendiente).
    • ('product_id', '!=', False): Asegura que la línea tenga un producto asociado.
  • Filtros Dinámicos (del usuario):
    • Se aplican filtros de fecha, cliente y búsqueda de manera similar a get_sales_lines, pero sobre los campos del pedido (order_id).

Lógica de "Pendiente"

Una línea se considera "pendiente" si su campo qty_to_invoice es mayor que cero. Este campo es calculado por Odoo y representa la cantidad que aún falta por facturar. Para pedidos en estado credit, este valor se calcula manualmente como product_uom_qty - qty_invoiced.

Campos y Datos Relacionados

El proceso es análogo al de ventas:

  1. Se obtienen las líneas de pedido de venta (sale.order.line) que cumplen con el dominio.
  2. Se filtran aquellas con qty_to_invoice > 0.
  3. Se recolectan los IDs de los pedidos (order_id), productos (product_id) y clientes (partner_id).
  4. Se realizan consultas secundarias a los modelos sale.order, product.product y res.partner para obtener los detalles completos.
  5. Se ensambla la estructura de datos final, calculando el total_pendiente para cada línea.

Campos de Fecha en Pedidos Pendientes

Importante:
  • fecha: Se obtiene del campo date_order de Odoo, que representa la fecha de confirmación del pedido. Este es el dato que aparece como "Fecha Pedido" en el dashboard y en las exportaciones.
  • commitment_date: Es un campo separado que indica la fecha comprometida de entrega al cliente. Aparece en las exportaciones en la columna "Fecha de entrega".
  • La tabla en el dashboard muestra la fecha de confirmación (date_order), mientras que la exportación incluye ambas fechas para una planificación más completa.

Sistema de Caché

Para mejorar el rendimiento y reducir la carga en el servidor de Odoo, la aplicación implementa un sistema de caché en memoria, especialmente en la ruta más consultada y costosa: el Dashboard Principal (/dashboard).

1. Propósito del Caché

La carga del dashboard principal implica múltiples y complejas consultas a Odoo para obtener ventas, pedidos pendientes y datos relacionados. Estas operaciones pueden ser lentas. El caché almacena el resultado de una petición ya procesada durante un tiempo determinado. Si otro usuario (o el mismo) realiza la misma petición dentro de ese tiempo, la aplicación devuelve la respuesta guardada en lugar de volver a consultar y procesar todo desde cero.

  • Mejora la velocidad: Las cargas subsiguientes de la página son casi instantáneas.
  • Reduce la carga de la API: Disminuye drásticamente el número de llamadas a Odoo.

2. Implementación (app.py)

Se utiliza la librería Flask-Caching. La configuración se define al inicio de app.py: 

# --- Configuración de Caché ---
cache_config = {
    "CACHE_TYPE": "SimpleCache",  # Almacenamiento en memoria simple
    "CACHE_DEFAULT_TIMEOUT": 600  # 10 minutos de caché por defecto
}
cache = Cache(config=cache_config)
cache.init_app(app)

Luego, se aplica un decorador a la ruta del dashboard: 

@app.route('/dashboard', methods=['GET', 'POST'])
@cache.cached(timeout=600, query_string=True)
def dashboard():
    # ... lógica de la función ...

Análisis del Decorador @cache.cached

  • timeout=600: Especifica que el resultado de esta ruta se almacenará en caché durante 600 segundos (10 minutos). Después de este tiempo, la primera petición que llegue volverá a ejecutar la función y renovará el caché.
  • query_string=True: Este es un parámetro crucial. Le indica al sistema de caché que cree una entrada de caché diferente para cada combinación de parámetros en la URL. Por ejemplo:
    • /dashboard tendrá su propio caché.
    • /dashboard?cliente_id=123 tendrá un caché completamente separado.
    • /dashboard?cliente_id=456 tendrá otro caché distinto.
    Esto asegura que cuando un usuario filtra por un cliente específico, vea los datos correctos para ese cliente y no los datos cacheados de "Todos los clientes".
Importante: El caché se invalida automáticamente después del `timeout`. Si se necesita forzar una actualización de datos antes de tiempo, sería necesario reiniciar la aplicación Flask.

Seguridad y Autenticación

La aplicación implementa un sistema de autenticación robusto basado en Google OAuth 2.0 con verificación adicional de lista blanca (whitelist) de usuarios autorizados.

1. Autenticación OAuth2 con Google

El sistema utiliza el protocolo OAuth 2.0 de Google para autenticar usuarios de forma segura, sin manejar contraseñas directamente.

Configuración de Credenciales OAuth2

Las credenciales de OAuth2 se configuran en el archivo .env: 

# OAuth2 Google
GOOGLE_CLIENT_ID=tu-client-id.apps.googleusercontent.com
GOOGLE_CLIENT_SECRET=tu-client-secret
PRODUCTION_URL=https://tu-dominio.com

Estas credenciales se obtienen desde la Google Cloud Console al crear un proyecto OAuth 2.0 (tipo: Web Application).

Flujo de Autenticación OAuth2

  1. Usuario accede al sistema: Al visitar la raíz (/), se redirige a /login.
  2. Inicio de OAuth2: El usuario hace clic en "Iniciar sesión con Google", lo que activa la ruta /login/google.
  3. Redirección a Google: La aplicación genera un URL de autorización usando google_auth_oauthlib.flow.Flow y redirige al usuario a la página de login de Google.
  4. Usuario autoriza: El usuario selecciona su cuenta de Google y autoriza el acceso solicitado (perfil básico: nombre y email).
  5. Callback de Google: Google redirige de vuelta a la aplicación en /login/callback con un código de autorización.
  6. Intercambio de tokens: La aplicación intercambia el código de autorización por un id_token.
  7. Verificación del token: Se verifica el id_token usando google.oauth2.id_token.verify_oauth2_token() para extraer la información del usuario (email, nombre).
  8. Validación de usuario autorizado: Se verifica que el email del usuario esté en la lista blanca (allowed_users.json).
  9. Creación de sesión: Si el usuario está autorizado, se crea una sesión de Flask almacenando username (email) y user_info en session.
  10. Redirección al dashboard: El usuario es redirigido a /dashboard.

Implementación en app.py

@app.route('/login/google')
def login_google():
    """Inicia el flujo de autenticación OAuth2 con Google"""
    try:
        # Crear flujo OAuth2
        flow = Flow.from_client_config(
            client_config=GOOGLE_CLIENT_CONFIG,
            scopes=['openid', 'email', 'profile'],
            redirect_uri=redirect_uri
        )
        
        # Generar URL de autorización
        authorization_url, state = flow.authorization_url(
            access_type='offline',
            include_granted_scopes='true',
            prompt='select_account'
        )
        
        # Guardar state en sesión (CSRF protection)
        session['state'] = state
        return redirect(authorization_url)
    except Exception as e:
        flash('Error al iniciar sesión con Google', 'danger')
        return redirect(url_for('login'))

@app.route('/login/callback')
def login_callback():
    """Maneja la respuesta de Google OAuth2"""
    try:
        # Verificar state (CSRF protection)
        if session.get('state') != request.args.get('state'):
            flash('Estado de sesión inválido', 'danger')
            return redirect(url_for('login'))
        
        # Obtener y verificar el id_token
        flow = Flow.from_client_config(...)
        flow.fetch_token(authorization_response=request.url)
        credentials = flow.credentials
        id_info = google.oauth2.id_token.verify_oauth2_token(
            credentials.id_token,
            google.auth.transport.requests.Request(),
            GOOGLE_CLIENT_ID
        )
        
        # Extraer información del usuario
        user_email = id_info.get('email')
        user_name = id_info.get('name')
        
        # Validar contra whitelist
        if user_email not in allowed_emails:
            flash('No autorizado', 'danger')
            return redirect(url_for('login'))
        
        # Crear sesión
        session['username'] = user_email
        session['user_info'] = {'name': user_name, 'email': user_email}
        
        return redirect(url_for('dashboard'))
    except Exception as e:
        flash('Error en autenticación', 'danger')
        return redirect(url_for('login'))

Seguridad OAuth2

  • CSRF Protection: Se utiliza el parámetro state para prevenir ataques CSRF.
  • HTTPS Required: En producción, OAuth2 requiere HTTPS. Para desarrollo local, se activa OAUTHLIB_INSECURE_TRANSPORT=1.
  • Token Verification: El id_token se verifica criptográficamente para garantizar su autenticidad.
  • Scopes Mínimos: Solo se solicitan los permisos necesarios: openid, email, profile.

2. Lista Blanca de Usuarios (Whitelist)

Incluso con una autenticación OAuth2 exitosa, solo los usuarios cuyo correo electrónico esté en la lista blanca podrán acceder a la aplicación.

  • Archivo de Configuración: La lista de usuarios autorizados se gestiona en el archivo allowed_users.json, ubicado en la raíz del proyecto.
  • Formato del Archivo: El archivo contiene una clave "allowed_emails" con una lista de los correos electrónicos permitidos. 
    {
    "allowed_emails": [
        "usuario1@agrovetmarket.com",
        "gerente.ventas@agrovetmarket.com"
    ]
}
  • Flujo de Validación:
    1. El usuario completa la autenticación OAuth2 con Google.
    2. app.py obtiene el correo electrónico del usuario del id_token.
    3. Lee el archivo allowed_users.json.
    4. Comprueba si el correo del usuario está en la lista allowed_emails.
    5. Si el usuario está en la lista, se le concede el acceso y se crea la sesión. Si no, se le muestra un mensaje de "No autorizado" y se le deniega la entrada.

3. Gestión de Sesiones

Flask utiliza sesiones basadas en cookies firmadas para mantener el estado de autenticación del usuario.

  • SECRET_KEY: Las sesiones se firman con la SECRET_KEY definida en .env para prevenir manipulación.
  • Datos de Sesión: Se almacenan username (email) y user_info (nombre, email) en la sesión.
  • Protección de Rutas: Todas las rutas protegidas verifican if 'username' not in session: y redirigen a /login si el usuario no está autenticado.
  • Logout: La ruta /logout limpia la sesión con session.clear() y redirige a /login.

4. Seguridad OWASP Top 10

La implementación de Google OAuth 2.0 en esta aplicación mitiga varios riesgos críticos del OWASP Top 10 (2021), que es el estándar de la industria para identificar las vulnerabilidades de seguridad web más críticas.

Protecciones Implementadas

Mitigación de Amenazas OWASP:
  • A01:2021 – Broken Access Control
    • ✅ OAuth 2.0: Autenticación centralizada con Google previene acceso no autorizado
    • ✅ Whitelist: Sistema de doble verificación (OAuth + lista de usuarios permitidos)
    • ✅ Verificación de sesión: Todas las rutas protegidas validan session['username']
    • ✅ Tokens firmados: Flask sessions con SECRET_KEY previene manipulación de sesión
  • A02:2021 – Cryptographic Failures
    • ✅ HTTPS obligatorio: OAuth 2.0 requiere HTTPS en producción para transmisión segura
    • ✅ Token verification: Verificación criptográfica del id_token con claves públicas de Google
    • ✅ Sin almacenamiento de contraseñas: No se manejan ni almacenan contraseñas (delegadas a Google)
    • ✅ SECRET_KEY: Clave secreta fuerte para firmar cookies de sesión
  • A03:2021 – Injection
    • ✅ Validación de entrada: OAuth tokens validados criptográficamente antes de uso
    • ✅ Escape automático: Jinja2 escapa automáticamente variables en templates
    • ✅ Consultas parametrizadas: XML-RPC de Odoo usa parámetros, no concatenación de strings
  • A04:2021 – Insecure Design
    • ✅ Autenticación robusta: Uso de estándar OAuth 2.0 en lugar de implementación custom
    • ✅ Separación de responsabilidades: Autenticación delegada a Google (expertos en seguridad)
    • ✅ CSRF protection: Parámetro state valida autenticidad de callbacks
  • A05:2021 – Security Misconfiguration
    • ✅ Variables de entorno: Credenciales en .env, nunca en código fuente
    • ✅ Scopes mínimos: Solo permisos necesarios (openid, email, profile)
    • ✅ Modo debug desactivado: app.run(debug=False) en producción
  • A07:2021 – Identification and Authentication Failures
    • ✅ MFA heredado: Si el usuario tiene 2FA en Google, se aplica automáticamente
    • ✅ Detección de anomalías: Google detecta accesos sospechosos antes de otorgar tokens
    • ✅ Session management: Sesiones seguras con cookies firmadas y HTTPOnly
    • ✅ Logout completo: session.clear() invalida completamente la sesión

Referencias de Seguridad

  • OWASP Top 10 (2021): https://owasp.org/www-project-top-ten/
  • Google Identity Platform: https://developers.google.com/identity
  • OAuth 2.0 RFC 6749: https://datatracker.ietf.org/doc/html/rfc6749
  • OWASP Authentication Cheat Sheet: https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html
Nota de Seguridad:

La combinación de Google OAuth 2.0 (autenticación delegada a un proveedor de identidad de confianza) + Whitelist (control de acceso adicional) proporciona una defensa en profundidad que protege contra la mayoría de amenazas comunes de autenticación y autorización.

Google mantiene activamente sus sistemas de seguridad, aplica parches rápidamente a vulnerabilidades, y proporciona protecciones avanzadas como detección de bots, análisis de comportamiento, y prevención de ataques de fuerza bruta - beneficios que heredamos automáticamente al usar su plataforma.

5. Configuración en Google Cloud Console

Para que OAuth2 funcione correctamente, es necesario configurar el proyecto en Google Cloud:

  1. Crear un proyecto en Google Cloud Console.
  2. Habilitar Google+ API o People API.
  3. Crear credenciales OAuth 2.0:
    • Tipo: Web Application
    • Authorized JavaScript origins: https://tu-dominio.com
    • Authorized redirect URIs: https://tu-dominio.com/login/callback
  4. Copiar Client ID y Client Secret al archivo .env.
  5. (Opcional) Configurar la pantalla de consentimiento OAuth con información de la aplicación.
Importante:
  • En desarrollo local, usar http://localhost:5000 como Authorized redirect URI.
  • En producción, SIEMPRE usar HTTPS.
  • La SECRET_KEY debe ser única y secreta (no compartir en repositorios públicos).
  • Mantener GOOGLE_CLIENT_SECRET confidencial (usar variables de entorno en producción).

Integración con Supabase

La aplicación utiliza Supabase (PostgreSQL) como base de datos para gestionar toda la información relacionada con las metas de venta y la composición de los equipos comerciales. La clase SupabaseManager (en desarrollo) encapsulará toda la lógica de esta interacción.

1. Autenticación y Configuración

La conexión se establece utilizando las credenciales de Supabase.

  • Variables de Entorno: Se requieren las credenciales de conexión en el archivo .env:
    • SUPABASE_URL: URL del proyecto Supabase
    • SUPABASE_KEY: API Key del proyecto
  • Cliente Supabase: Se utiliza la librería supabase-py para interactuar con la base de datos PostgreSQL.

2. Estructura de Datos en Supabase

El sistema utiliza tablas relacionales en PostgreSQL para almacenar las metas y configuraciones:

  • metas_clientes: Almacena las metas anuales por cliente. 
    - id (primary key)
- año (integer)
- cliente_id (integer)
- cliente_nombre (text)
- meta (numeric)
- created_at / updated_at
  • metas_lineas: Almacena las metas mensuales por línea comercial. 
    - id (primary key)
- año (integer)
- mes (integer)
- linea_comercial (text)
- meta (numeric)

3. Flujo de Lectura y Escritura

Las rutas de Flask como /meta y /metas_vendedor utilizan el SupabaseManager para gestionar estos datos.

  1. Lectura (GET): Cuando un usuario visita una página de metas, app.py realiza consultas SQL a través del cliente Supabase (ej. supabase.table('metas_clientes').select('*').eq('año', 2025).execute()). Los datos se devuelven como objetos Python listos para usar.
  2. Procesamiento: La aplicación utiliza estos datos para rellenar los formularios y tablas que ve el usuario.
  3. Escritura (POST): Cuando el usuario guarda cambios, app.py recolecta los datos del formulario y realiza operaciones INSERT, UPDATE o UPSERT en las tablas correspondientes de Supabase.

4. Ventajas de Supabase

  • Relacional: Permite consultas SQL complejas y relaciones entre tablas
  • Escalable: Maneja grandes volúmenes de datos eficientemente
  • Seguro: Políticas de seguridad RLS (Row Level Security) integradas
  • Real-time: Capacidad de suscripciones en tiempo real (para futuras mejoras)
  • Backup automático: Respaldos automáticos de la base de datos

Lógica de los Dashboards

Esta sección detalla cómo se procesan y calculan los datos para las vistas principales del dashboard, que son las más complejas de la aplicación.

1. Dashboard Internacional (/dashboard)

Esta vista consolida las ventas y pedidos pendientes del canal internacional para el año en curso (o un rango de fechas específico si se implementara).

Flujo de Datos

  1. Obtención de Datos Crudos: La función llama a data_manager.get_sales_lines() para obtener todas las líneas de facturas del período y a data_manager.get_pending_orders() para los pedidos pendientes. Ambas llamadas se pueden filtrar por un partner_id (cliente).
  2. Obtención de Metas: Llama a gs_manager.read_metas_por_cliente() para obtener las metas anuales de todos los clientes.
  3. Cálculo de KPIs Principales:
    • Ventas Totales del Año: Se calcula sumando el campo amount_currency de todos los registros devueltos por get_sales_lines.
    • Total Por Facturar: Se calcula sumando el campo total_pendiente de todos los registros de get_pending_orders.
    • Meta y Brecha: La meta se calcula sumando las metas de los clientes (o del cliente seleccionado). La brecha es la diferencia entre la proyección (Ventas + Por Facturar) y esta meta real.
  4. Agregación para Tabla Principal:
    • Se inicializa un diccionario ventas_por_linea. Se itera sobre los datos de ventas y se agrupan las sumas de amount_currency por el nombre de la línea comercial internacional (commercial_line_international_id).
    • De forma similar, se itera sobre los datos pendientes y se agrupan en por_facturar_por_linea.
    • Finalmente, se combinan ambos diccionarios para construir la tabla que el usuario ve en pantalla.
    • Para la columna "Meta" de la tabla, se suman las metas de los clientes correspondientes a cada línea comercial.
  5. Preparación de Datos para Gráficos:
    • Top Productos y Gráficos de Pastel: Se agrupan las ventas por nombre de producto, clasificación farmacológica, etc., para alimentar los gráficos.
    • Análisis Jerárquico (Drilldown): Se utiliza la librería pandas para convertir los datos de ventas en un DataFrame. Luego, se realizan agrupaciones sucesivas (groupby) para construir la estructura de datos jerárquica que espera el gráfico ECharts. La jerarquía actual es:
      1. clasificacion_farma_nombre
      2. forma_farma_nombre
      3. via_admin_nombre
      4. linea_prod_nombre
      5. producto_nombre
    • Avance por Cliente (General): Se crean dos diccionarios, uno para agrupar las ventas facturadas por cliente y otro para los pendientes. Luego se unen para obtener el total de pedido (facturado + pendiente) y el avance para cada cliente. Este gráfico se oculta si se selecciona un cliente.
  6. Lógica para Cliente Seleccionado:
    • Cálculos Dinámicos: Si se detecta un partner_id en los filtros, los cálculos de KPIs y de la tabla principal se rehacen usando solo los datos de ventas y metas de ese cliente específico.
    • Visualización de Gráficos: En el frontend, el JavaScript detecta la presencia de selectedClienteId y alterna la visibilidad: oculta el gráfico general de clientes y muestra la tarjeta de avance y el gráfico de detalle de pedidos para el cliente seleccionado.
    • Tabla de Productos del Cliente: Funcionalidad avanzada implementada en dashboard_clean.html:
      • Activación: Botón "📦 Productos" en el gráfico de Avance por Pedido que ejecuta toggleProductosCliente()
      • Procesamiento de datos: Función procesarProductosCliente() que:
        • Itera salesDataAll y pendingDataAll (inyectados desde backend)
        • Filtra por partner_id === selectedClienteId
        • CRÍTICO: Usa Map con productoKey = productoCodigo (campo default_code o codigo_odoo) — NUNCA nombre del producto
        • Acumula cantidades y montos (facturado + pendiente) por código único
        • Pobla selector dinámico de líneas comerciales
      • Filtrado: Función filtrarProductosCliente() aplica filtros de estado y línea comercial, ordena por monto descendente
      • Paginación: Sistema completo con:
